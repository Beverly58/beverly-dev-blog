## Translating Design into Data (LO1)

This week focused on translating my **Fandom Passport wireframe** into structured backend logic using a **Data Definition Document (DDD)**.

Following the workshop instructions, I identified key **entities** from my interface, including:
- Users (fans)
- Stamps (achievements)
- Tasks (actions that generate stamps)
- User_Stamps (records of collected stamps)
- Passport (progress container)

Each entity was broken down into attributes with clear descriptions and example values. This process helped clarify what data the system must store and how each feature is supported technically.

Importantly, I avoided vague attributes such as “list of stamps” and instead separated them into proper relational structures, which aligns with database normalization principles.

---

## Structuring Relationships with ERD Thinking (LO2)

Based on the DDD, I then mapped relationships between entities to prepare for an **Entity Relationship Diagram (ERD)**.

### Key Relationships

- A **User** can earn many **Stamps**
- A **Stamp** can be earned by many **Users**
→ This forms a **many-to-many relationship**

To resolve this, I introduced a junction entity:

- **User_Stamps**
  - connects Users and Stamps
  - records timestamp and status of each achievement

This follows the standard database pattern discussed in class, where many-to-many relationships must be resolved through a **junction table with a composite key**.

## Designing for System Logic, Not Just UI (LO3)

One key insight this week is that **data design shapes interaction design**.

For example:
- The passport “grid” UI is not just visual — it directly reflects database structure
- Empty stamp slots correspond to missing rows in the User_Stamps table
- Filled stamps are dynamically rendered based on database queries

By planning this structure early, I ensure that future implementation using **MojoJS + SQLite + HTMX** can efficiently:
- Query user progress
- Update stamps dynamically
- Scale when new stamps or events are added

This also supports the **DRY principle**, as new stamps only need to be added once in the database to become available across all users.

## Reflection

Compared to earlier weeks where I focused mainly on interface design, this stage required a shift toward **system-level thinking**.

The DDD and ERD process made me realise that:
- Good interaction design depends on solid data structure
- Every UI element must be backed by a clear data model
- Designing relationships early prevents technical limitations later

This week bridges the gap between concept and implementation, forming the foundation for building the full prototype in upcoming weeks.
